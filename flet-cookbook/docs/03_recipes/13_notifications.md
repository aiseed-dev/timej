# レシピ#3-13: 通知（プッシュ）とローカルなお知らせ

「新着があったら知らせたい」の実装は、**アプリが開いている間**と
**閉じている間**で難易度が大きく違う。

## アプリ内のお知らせ（開いている間）

開いている間の通知は簡単。`SnackBar` や画面内のバッジで十分:

```python
page.open(ft.SnackBar(ft.Text("新着が3件あります")))
page.update()
```

定期取得（[2-2](../02_core_concepts/02_state.md) の use_effect + タイマー）で
差分を引き、新着があればお知らせ一覧に足す。閉じた輪の中の連絡なら、
**アプリ内メッセージ＋既読管理**でメールを置き換えられる（開封時刻を
記録すれば、メールにできない到達確認ができる）。

## プッシュ通知（閉じている間）＝ FCM

アプリが閉じている・バックグラウンドのときに届けるには、OS のプッシュ
（Firebase Cloud Messaging, FCM）が要る。Flet 単体には無いので**拡張**を
使う/作ることになる。

- **バックグラウンド表示**は FCM の `notification` メッセージに委ねる
  （OSが勝手に通知を出す。アプリのPythonは動いていない）
- **Python 側**が担うのは:
  - トークンの登録（この端末の宛先をサーバに送る）
  - フォアグラウンド受信（開いている間に来たデータの処理）
  - 起動時の initial message 処理（通知タップで開かれた時の初期画面）

!!! warning "iOS の FCM は難所"
    iOS は APNs 経由で、証明書・権限・バックグラウンドハンドラの設定が
    要る。ここは Flet でいちばん native に踏み込む部分。バージョン更新
    （0.85→0.86 のような）で壊れやすいので、**通知は最後に載せる**のが
    無難（アプリの主機能が固まってから）。

## トピック設計（誰に届けるか）

宛先を「トピック」で束ねると運用が楽:

- `org.<id>` … 組織の全告知
- `org.<id>.members` … 会員限定
- `org.<id>.<series>` … 系列（例: あるイベント）
- 個別（受付完了・締切リマインド）はトークン宛て直送

購読は利用者が設定画面で明示的に選ぶ。既定は「申込済みの系列」だけ、と
最小にしておくと、通知疲れを避けられる。
