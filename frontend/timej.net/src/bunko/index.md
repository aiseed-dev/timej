# 文庫（bunko） — 青空文庫をUnicode構造化データに

文庫は、青空文庫の正本（注記付きテキスト）を、**構造化されたUnicodeデータに
変換するPythonライブラリ（pybunko）**と、そこから生成した**データ資産**です。
外字を画像ではなく実Unicode文字に解決するので、選択・検索・読み上げが
そのまま効きます。[AISeed工房（pykobo）](../kobo/)の土台でもあります。

## 思想 — 正本はテキスト、器は後から生やす

青空文庫が四半世紀を生き延びたのは、**正本が静的なテキストファイル**
だったからです。テキストはミラーでき、パースでき、四半世紀前のファイルが
今日そのまま読めます。だから文庫は、機能をサーバに足すのではなく、
**テキストから読者の手元に生やす**——という青空文庫自身の強さをなぞります。

一次表現として残すべきは、器（HTMLやXHTML）ではなく、**外字まで解決した
構造化Unicodeデータ**（Document）だと考えます。それさえあれば、表示も
電子書籍化も朗読も、あとからいくらでも生成できます。

## できること

正本 → **Document（唯一の中間表現）** → 各出力、という一本道です。

- **変換** — 注記付きテキストを解析し、外字・アクセントを実Unicode文字に
  解決したDocumentへ。ルビは「読みデータ」として保持（TTS・検索に効く）
- **出力** — JSON / SQLite（書架DB）/ EPUB / 公式XHTML再現 / 読み上げ
  テキスト / Markdown / Parquet
- **データ資産** — 全17,335作品・981作家のメタデータ（SQLite 1ファイル）、
  代表作の本文・図書カード、JIS X 0213全外字を収めた明朝フォント
- **外字フォント** — 真の外字4,330字を軽量サブセット（約2.8MB）に。
  アプリに1フォント同梱すれば全外字を実文字表示

## 原則

- **公式サーバー（aozora.gr.jp）には触れない** — 取得はGitHubミラーの静的
  ファイルのみ、必ずキャッシュ経由（二度目からオフライン）
- **ゼロ依存コア** — 本体は標準ライブラリのみ。EPUB・組版・フォント等の
  重い機能はオプション
- **著作権存続作品はデフォルト除外**

## 位置づけ

文庫はライブラリとデータ資産に純化しています。「書く・組む」はエディタ
（[工房 / pykobo](../kobo/)）が、「縦書き組版」はpywashiが担い、文庫は
その全員が使う**変換とデータの土台**です。
