# 文庫（bunko） — 形式から構造へ、テキストを一次表現に

これは [AIネイティブな仕事の作法](https://aiseed.dev/ai-native-ways/)（構造）の、
**構造化とデータ**の実践です。文庫は、青空文庫の正本（注記付きテキスト）を
**構造化されたUnicodeデータに変換するPythonライブラリ（pybunko）**と、そこから
生成したデータ資産。[AISeed工房（pykobo）](../kobo/)の土台でもあります。

## 作法 — 形式ではなく、構造を持つ

作法の出発点は「**見た目の形式ではなく、意味の構造を持つ**」ことです。
青空文庫の外字注記や画像外字は「形式」ですが、文庫はそれを**外字まで解決
した構造化データ（Document）**にします。

- 外字は画像でなく**実Unicode文字**に解決 → 選択・検索・読み上げがそのまま効く
- ルビは飾りでなく**読みデータ**として保持 → TTS・検索に使える
- 正本 → **Document（唯一の中間表現）** → 各出力、という一本道

構造さえ持てば、表示も電子書籍化も朗読も後から生成できます。

## 作法 — テキストが一次表現、器は生やす

青空文庫が四半世紀を生き延びたのは、**正本が静的なテキスト**だったから。
テキストはミラーでき、パースでき、四半世紀前のファイルが今日そのまま
読めます。文庫はこの強さをなぞり、**残すべき一次表現をテキスト由来の構造**に
置きます。

- **中身（残すもの）** … Document（JSON）。これが一次データ資産
- **外枠（生やすもの）** … HTML / EPUB / 公式XHTML / PDF / 読み上げ

器はいくらでも作り直せる派生物なので、投資は中身に集める。工房もpywashiも、
この同じ中身を使って各自の外枠を生やします。

## 作法 — データは自己記述の標準形で

データの置き場は、位置依存のCSVではなく、**自己記述でAIが扱える標準形**に。

- 書架メタは **SQLite**（全17,335作品/981作家を1ファイル・検索と結合が速い）
- 本文・図書カードは正規化しすぎず **JSON** のまま列に持つ
- 全作品コーパスは研究・NLP向けに **Parquet** で列指向エクスポート

ツールでなくデータ形式に賭けるので、部品は差し替え可能・データはツールより
長生きします。

## 原則

- **ゼロ依存コア** — 本体は標準ライブラリのみ。EPUB・組版・フォント等は
  オプション（段階的スケーリング）
- **公式サーバー（aozora.gr.jp）には触れない** — 取得はGitHubミラーの静的
  ファイルのみ、必ずキャッシュ経由（二度目からオフライン）
- **著作権存続作品はデフォルト除外**
