## 🎯 トークン節約の基本原則

### 原則1: 対象を明示して、範囲を絞る

**❌ 悪い例：**
```
「アプリ全体を見て、ボタンの色を赤に変更してください」
→ Claude Codeが不要なファイルまで探索・読み込みする
```

**✅ 良い例：**
```
「lib/home_page.dart の送信ボタンを赤色にしてください」
→ 必要なファイルだけを読んで処理
```

**削減効果：** 90%以上のトークン節約

---

### 原則2: 目的に合った頼み方をする

| 頼み方 | トークン消費 | 使うべき場面 |
|--------|------------|------------|
| **箇所を指定した小さな修正** | ⭐ 最小 | 1ファイル・1箇所の変更 |
| **「変更せず説明して」** | ⭐⭐ 小〜中 | 質問・相談・エラー調査 |
| **機能単位の実装依頼** | ⭐⭐⭐⭐⭐ 最大 | 複数ファイルの変更・新規作成 |

**基本ルール：**
- 小さな変更 → ファイルと箇所を指定して依頼
- 質問・相談 → 「コードは変更せず、説明だけしてください」と添える
- 大規模な実装 → まず計画を出させて、段階ごとに実装

---

### 原則3: モデルを使い分ける

`/model` でいつでも切り替えられます。

**Opus 4.5（高コスト）を使うべき場面：**
- 複雑なアーキテクチャ設計
- 大規模なリファクタリング
- 難しいバグの調査
- 初めての技術への挑戦

**Sonnet 4.5（低コスト）で十分な場面：**
- 簡単なUI実装
- 定型的なコード生成
- エラーメッセージの解読
- 小さな修正・調整

**削減効果：** 80%のコスト削減（同じ作業でも）

## 🛠️ 実践テクニック

### テクニック1: ファイル名を明示する

**❌ 無駄なコンテキスト：**
```
「アプリ全体を見て、home_page.dartのボタンを修正してください」
→ プロジェクト全体を探索（数千〜数万トークン）
```

**✅ 必要なファイルだけ指定：**
```
「lib/home_page.dart のボタンを修正してください」
→ 1ファイルだけ読み込み（数百トークン）
```

**削減効果：** 90%以上のトークン節約

---

### テクニック2: 段階的に質問する

**❌ 一度に全てを聞く：**
```
「Flutterの状態管理について、setState、Provider、Riverpod、BLoC、GetXの
違いを詳しく説明してください。それぞれのメリット・デメリット、使い分け、
サンプルコード、ベストプラクティス、アンチパターンも教えてください」
→ 膨大な回答（数千トークン）
```

**✅ 必要なことだけ聞く：**
```
「FlutterでsetStateとProviderの違いを簡潔に教えてください」
→ 必要な情報だけ（数百トークン）
```

**削減効果：** 70〜80%のトークン節約

---

### テクニック3: プロンプトを具体的にする

**❌ 曖昧な指示（往復が増える）：**
```
「TODOアプリを作ってください」

Claude: 「どのような機能が必要ですか？」
あなた: 「TODOの追加と削除」
Claude: 「データはどこに保存しますか？」
あなた: 「SharedPreferencesで」
→ 3往復（トークン3倍）
```

**✅ 具体的な指示（1回で完了）：**
```
「TODOアプリを作ってください。機能：TODOの追加・削除・完了チェック。
データ保存：SharedPreferences。UI：Material Design。」

Claude: [実装を開始]
→ 1往復のみ
```

**削減効果：** 60〜70%のトークン節約

---

### テクニック4: 小さく実装して確認する

**❌ 大規模な実装を一度に依頼：**
```
「TODOアプリを完全に実装してください。ホーム画面、詳細画面、設定画面、
データベース、API通信、認証、通知、テーマ切り替え、多言語対応も。」
→ 実装後に問題発覚 → 大規模な修正が必要（トークン大量消費）
```

**✅ 小さく作って確認：**
```
1. 「TODOのホーム画面だけ作成してください」 → 確認 → OK
2. 「SharedPreferencesでの保存機能を追加してください」 → 確認 → OK
3. 「削除機能を追加してください」 → 確認 → OK
```

**削減効果：** 失敗時の無駄なトークンを削減

---

### テクニック5: エラーは最小限の情報で相談

**❌ 全ての情報をコピペ：**
```
[エラーログ全文をコピペ（500行）]
[該当ファイル全文をコピペ（300行）]
[pubspec.yaml全文をコピペ（100行）]
「このエラーを修正してください」
→ 900行のコンテキスト（数千トークン）
```

**✅ エラーの核心だけ送る：**
```
「以下のエラーが出ました：
The method 'setState' isn't defined for the class 'MyWidget'

lib/home_page.dart の50行目付近を見てください」
→ 必要最小限（数百トークン）
```

なお、Claude Codeは自分で `flutter analyze` を実行してエラーを取得できるので、
「flutter analyze を実行して、出たエラーを直してください」という頼み方も有効です。

**削減効果：** 80〜90%のトークン節約

---

### テクニック6: /clear で会話をリセットする

**会話の履歴は累積する：**
```
1回目の質問: 100トークン
2回目の質問: 100 + 前回の履歴200トークン = 300トークン
3回目の質問: 100 + 前回の履歴500トークン = 600トークン
```

**新しいトピックは新しい会話で：**
- タスクが終わったら `/clear` で会話をリセット
- 長くなった会話は `/compact` で要約して続行
- 恒常的な前提は会話ではなく `CLAUDE.md` に書く（毎回自動で読み込まれる）

**削減効果：** 50%以上のトークン節約（長い対話の場合）

## 🎓 トークン節約のチェックリスト

### 実装前のチェック

- [ ] 対象のファイル・箇所を明示したか？
- [ ] プロンプトは具体的か？（往復を減らせるか？）
- [ ] Sonnet 4.5で十分な作業か？（Opusが必要か？）
- [ ] 恒常的な前提は CLAUDE.md に書いてあるか？

### 実装中のチェック

- [ ] 小さく実装して確認しているか？（大規模な失敗を避けられるか？）
- [ ] 終わったタスクの会話を引きずっていないか？（/clear）
- [ ] エラー情報は必要最小限か？（全文コピペしていないか？）

### 実装後のチェック

- [ ] 次回のために、効率的だった方法をメモしたか？（CLAUDE.md に足すのも良い）
- [ ] 同じ質問を何度もしていないか？（よくある質問は記録しておく）

## 🚀 実践的な節約ワークフロー

### ワークフロー1: 新機能の実装

```
1. 【Web版Claude または Claude Code + Sonnet】設計を相談（低コスト）
   ↓
2. 【Claude Code + Sonnet】機能単位で実装（中コスト）
   ↓
3. 【Claude Code + Sonnet】箇所を指定して微調整（低コスト）
   ↓
4. 【Claude Code + Opus】複雑な問題の相談（必要時のみ）
```

---

### ワークフロー2: バグ修正

```
1. エラーログを確認
   ↓
2. 【Sonnet】「このエラーの原因を、コードを変更せずに説明して」（低コスト）
   ↓
3. 【Sonnet】該当ファイルを指定して修正（低コスト）
   ↓
4. （解決しない場合）【Opus】深い調査
```

## 📝 まとめ

**トークン節約の3大原則：**

1. **対象を明示して範囲を絞る**
   - 小さな変更 → ファイルと箇所を指定
   - 質問・相談 → 「変更せず説明して」
   - 大規模実装 → 計画を出させて段階ごとに

2. **必要最小限のコンテキスト**
   - ファイル名を明示する
   - エラーは核心だけ送る
   - /clear で会話をリセット、前提は CLAUDE.md へ

3. **モデルを使い分ける**
   - 日常作業 → Sonnet 4.5（80%）
   - 複雑な作業 → Opus 4.5（20%）

**これらのテクニックで、無駄なコストを抑えながら、生産性を最大化できます。**

効率的なAI活用は、開発スキルの一部です。賢く使って、コストと生産性の両方を最適化しましょう。
